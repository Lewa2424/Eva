Текущий статус проекта Eva 2.0

Это рабочая шпаргалка выполнения проекта.
Сюда заносится всё, что уже сделано по факту.
Файл должен пополняться после каждого существенного завершённого шага.

Связка документов проекта:
- _docs/Ева_2.0_чистовик_0.4.txt — основной план создания;
- PROJECT_STATUS.txt — живая шпаргалка выполнения и текущего состояния.

Что уже сделано

1. Создан базовый каркас проекта:
   - eva_core/
   - eva_code/
   - eva_test/
   - eva_utils/
   - eva_api/
   - reflection_log/

2. Подготовлена документационная основа:
   - есть чистовик-план создания Eva 2.0;
   - есть ARCHITECTURE.md с принципами модульности;
   - зафиксировано правило: один файл = одна роль.

3. Реализован минимальный входной контур MVP:
   - main.py загружает .env;
   - создаётся MemoryManager;
   - создаётся DecisionEngine;
   - тестовая цель сохраняется в БД;
   - запускается SelfMonitor.scan();
   - выполняется проверка подключения к БД.

4. Реализована базовая память через PostgreSQL:
   - schema.sql создаёт таблицы tasks, plans, code_versions, test_results, reflections;
   - schema_code_map.sql создаёт tables modules и dependencies;
   - MemoryManager умеет сохранять и читать основные сущности;
   - добавлена работа с modules и dependencies.

5. Реализован минимальный DecisionEngine:
   - run_once(goal) сохраняет цель как задачу в БД;
   - DecisionEngine пока не планирует и не маршрутизирует действия.

6. Реализован минимальный SelfMonitor:
   - сканирует все .py-файлы проекта;
   - исключает .venv, venv, __pycache__, .git, site-packages;
   - сохраняет файлы в таблицу modules;
   - обновляет last_seen_at;
   - считает content_hash для каждого файла;
   - определяет, изменился файл или нет.

7. Проверено, что отслеживание изменений работает:
   - если файл не менялся, обновляется только last_seen_at;
   - если файл менялся, обновляется content_hash;
   - Ева уже умеет видеть, какой именно файл изменился.

8. Реализован LLMProxy как единая точка доступа к LLM:
   - прямые вызовы модели из других модулей не предполагаются;
   - пока это инфраструктурный слой, а не часть рабочего цикла Евы.

9. Принято архитектурное решение по AST-слою:
   - не хранить структуру кода в modules.meta;
   - не создавать отдельные таблицы под каждый файл или директорию;
   - хранить структуру кода в отдельных общих таблицах по типу сущностей.

10. Реализован минимальный AST-слой:
   - schema_code_map.sql расширен таблицами code_entities и module_imports;
   - SelfMonitor научен разбирать новые и изменённые .py-файлы через ast;
   - добавлена гарантия первичного AST-разбора даже для неизменённых файлов, если снимка ещё нет;
   - извлекаются классы, функции, методы и импорты;
   - MemoryManager умеет заменять AST-снимок модуля в отдельных таблицах.

11. Docker временно отложен:
   - безопасная песочница и исполнение кода пока не реализуются;
   - текущий фокус — память, карта кода, AST-структура и архитектурная база.

12. Реализована автоматическая карта import-зависимостей:
   - SelfMonitor после сканирования пересобирает внутренние зависимости проекта;
   - зависимости строятся на основе уже сохранённых AST-импортов;
   - в dependencies записываются только внутренние связи между модулями Евы;
   - внешние библиотеки в dependencies не попадают.

13. Исправлен архитектурный перекос в MemoryManager:
   - разросшийся memory_manager.py разделён по зонам ответственности;
   - MemoryManager сохранён как единый фасад памяти;
   - реализация вынесена в отдельные файлы:
     - memory_tasks.py
     - memory_modules.py
     - memory_dependencies.py
   - текущий публичный API MemoryManager сохранён без поломки существующих вызовов.

На чём сейчас остановились

Сейчас завершены:
- первый рабочий вертикальный срез;
- второй слой самонаблюдения;
- первый слой реальной карты связей проекта.

цель -> запись в память -> скан проекта -> фиксация изменений файлов

То есть Ева уже умеет:
   - стартовать;
   - писать задачу в БД;
   - видеть свою файловую структуру;
   - замечать изменения в кодовой базе.

И в коде уже подготовлено:
   - извлечение AST-структуры изменённых файлов;
   - отдельное хранение сущностей кода и импортов.

И уже проверено на живой базе:
   - все 11 актуальных Python-файлов проекта получили AST-снимок;
   - в code_entities появились записи сущностей кода;
   - в module_imports появились записи импортов.
   - в dependencies появились реальные внутренние import-зависимости.
   - после разбиения MemoryManager проект по-прежнему запускается без ошибок.

Что ещё не сделано

- нет более глубоких связей уровня "кто кого вызывает";
- нет маршрутизации действий;
- нет полноценного цикла планирования;
- нет безопасной Docker-песочницы;
- нет реальной самомодификации.

Следующий логичный шаг

Следующий логичный шаг: начать строить более глубокие связи уровня "кто кого вызывает" и готовить основу для оценки влияния изменений.

Что должен дать этот шаг:
   - следующий уровень понимания физиологии кода;
   - основу для ответа на вопрос, что сломается при изменении функции или метода;
   - базу для будущих метрик критичности и архитектурной слабости.

Почему именно этот шаг

Потому что он:
   - логически продолжает уже готовую карту import-зависимостей;
   - не требует Docker;
   - не меняет архитектурный курс;
   - переводит Еву от уровня "вижу модули" к уровню "вижу поведение связей".

16. Реализован минимальный слой метрик критичности и хрупкости:
   - schema_code_map.sql расширен таблицами entity_metrics и module_metrics;
   - добавлен отдельный модуль memory_metrics.py для хранения и чтения метрик;
   - добавлен отдельный модуль criticality_analyzer.py для расчёта метрик по сущностям и модулям;
   - расчёт опирается на уже готовые entity_relations, dependencies, entity_impacts и module_impacts;
   - SelfMonitor теперь после синхронизации AST, связей и impact-слоя запускает отдельный метрический анализатор.

Какие метрики сейчас считает Ева:
   - по сущности:
     - direct_entity_impact_count
     - indirect_entity_impact_count
     - incoming_call_count
     - criticality_score
     - fragility_score
   - по модулю:
     - direct_module_impact_count
     - indirect_module_impact_count
     - incoming_dependency_count
     - criticality_score
     - fragility_score

Как сейчас считается score:
   - формулы простые и прозрачные;
   - criticality_score усиливает прямое влияние и входящие связи;
   - fragility_score даёт близкую, но немного мягче взвешенную оценку хрупкости;
   - на этом шаге не используется сложная теория графов и не вводятся неочевидные коэффициенты.

Что подтверждено на живой базе после внедрения:
   - main.py запускается без ошибок;
   - в entity_metrics сохранено 98 записей;
   - в module_metrics сохранено 20 записей;
   - можно получить top critical entities и top critical modules прямо из базы;
   - наиболее критичным модулем сейчас определяется eva_core/memory_manager.py.

Что теперь уже умеет Ева:
   - не только видеть последствия изменения;
   - но и ранжировать узлы по критичности и хрупкости;
   - выделять, какие функции и модули наиболее чувствительны для изменений.

Что этот слой пока сознательно не покрывает:
   - продвинутые графовые метрики;
   - историческое сравнение старого и нового состояния;
   - сравнение качества кода до и после изменения;
   - адаптивные веса формул.

Новый следующий логичный шаг:
   - добавить слой сравнения состояний кода "до / после";
   - научить Еву измерять, стало ли изменение лучше или хуже по структуре, влиянию и риску.
