# PROJECT_MAP

Актуальная карта проекта `Eva 2.0` по текущему состоянию файлов.

Назначение файла:
- быстро видеть общую структуру проекта;
- понимать, за что отвечает каждая директория и каждый файл;
- сверяться перед изменениями;
- замечать пробелы между планом и текущей реализацией.

Важно:
- это структурная карта проекта, а не журнал изменений;
- файл должен обновляться при создании, удалении или смене роли файлов;
- секреты и реальные значения из `.env` сюда не заносятся.

## 1. Корень проекта

Корневая директория: `Eva/`

Роль корня:
- хранит точку входа;
- хранит общие архитектурные документы;
- хранит настройки окружения;
- соединяет все подсистемы проекта в единую структуру.

### Файлы корня

#### `.env`
- Тип: локальный конфигурационный файл.
- Назначение: хранит реальные переменные окружения для локального запуска.
- Для чего нужен:
  - подключение к PostgreSQL;
  - доступ к LLM/API;
  - локальные параметры среды.
- Статус: локальный файл, не должен попадать в Git.
- Примечание: содержимое не документируется в этой карте по соображениям безопасности.

#### `.env.example`
- Тип: шаблон конфигурации.
- Назначение: показывает, какие переменные окружения нужны проекту.
- Для чего нужен:
  - быстрый старт на новой машине;
  - документирование обязательных переменных;
  - безопасная замена `.env` в репозитории.
- Статус: используется как публичный шаблон.

#### `.gitignore`
- Тип: служебный файл Git.
- Назначение: исключает из репозитория локальные и временные файлы.
- Для чего нужен:
  - не пушить `.env`;
  - не пушить `.venv`;
  - не пушить кэш и артефакты среды разработки.
- Статус: активный.

#### `AGENTS.md`
- Тип: файл правил работы.
- Назначение: фиксирует принципы архитектурной дисциплины и формат взаимодействия.
- Для чего нужен:
  - удерживать чистую архитектуру;
  - запрещать самовольные изменения без запроса;
  - задавать стиль анализа и внесения изменений.
- Статус: активный управляющий документ.

#### `ARCHITECTURE.md`
- Тип: архитектурный документ.
- Назначение: описывает целевую архитектуру `Eva 2.0`.
- Для чего нужен:

#### `llm_proxy.py`
- Тип: инфраструктурный Python-модуль.
- Назначение: единая точка доступа к LLM.

- Статус: инфраструктурный слой готов частично.

- Статус: реализован минимальный каркас.

#### `eva_core/memory_manager.py`
- Назначение: единая точка работы с PostgreSQL.
- Для чего нужен:
  - дать проекту стабильный единый интерфейс памяти;
  - скрыть внутреннее разбиение реализации по файлам.
- Что уже есть:
  - подключение через `POSTGRES_DSN`;
  - `ping()`;
  - короткий фасад поверх внутренних memory-модулей.
- Что ещё нет:
  - дальнейшего разбиения других крупных модулей проекта по той же дисциплине.
- Статус: короткий фасад, оставлен как единая точка памяти.

#### `eva_core/memory_tasks.py`
- Назначение: операции памяти для задач, планов, кодовых версий, тестов и рефлексии.
- Для чего нужен:
  - убрать лишнюю смешанную ответственность из `memory_manager.py`;
  - держать бизнес-операции памяти в отдельном модуле.
- Статус: активный внутренний модуль реализации.

#### `eva_core/memory_modules.py`
- Назначение: операции памяти для карты модулей, AST-снимков и импортов.
- Для чего нужен:
  - хранить структуру файлов и их внутреннее устройство;
  - обслуживать `SelfMonitor`.
- Статус: активный внутренний модуль реализации.

#### `eva_core/memory_dependencies.py`
- Назначение: операции памяти для зависимостей между модулями.
- Для чего нужен:
  - читать, заменять и пересобирать связи между модулями;
  - не смешивать зависимости с остальной памятью.
- Статус: активный внутренний модуль реализации.

#### `eva_core/self_monitor.py`
- Назначение: минимальное самонаблюдение Евы.
- Для чего нужен:
  - сканировать проект;
  - находить Python-файлы;
  - сохранять их в мета-карту;
  - отслеживать изменения по хэшу.
- Что уже есть:
  - поиск `.py`-файлов;
  - пропуск `.venv`, `venv`, `__pycache__`, `.git`, `site-packages`;
  - вычисление `SHA256`;
  - сравнение с `content_hash`;
  - обновление `last_seen_at`;
  - запись новых и изменённых файлов в таблицу `modules`;
  - AST-разбор новых и изменённых файлов;
  - первичный AST-разбор даже для неизменённых файлов, если AST-снимок ещё не создан;
  - извлечение классов, функций, методов и импортов;
  - запись AST-структуры в отдельные таблицы;
  - автоматическая пересборка внутренних import-зависимостей между модулями проекта.
- Что ещё нет:
  - анализа вызовов между функциями и методами;
  - полноценных метрик кода.
- Статус: готовы зрение, структурный анализ и первый слой реальной карты связей.

## 10. Что проект умеет на текущий момент

Текущий рабочий контур:

`цель -> запись в БД -> скан проекта -> фиксация изменений файлов`

То есть система уже умеет:
- стартовать из `main.py`;
- подключаться к PostgreSQL;
- создавать задачу через `DecisionEngine`;
- сканировать Python-файлы;
- сохранять карту файлов в БД;
- определять, какой файл изменился.

Дополнительно в коде уже подготовлено:
- разбирать новые и изменённые Python-файлы через AST;
- извлекать функции, классы, методы и импорты;
- хранить их в отдельных таблицах, а не в `modules.meta`.

И уже подтверждено на живой базе:
- все 11 актуальных Python-файлов проекта имеют AST-снимок;
- `code_entities` заполнена;
- `module_imports` заполнена.
- `dependencies` автоматически заполнена внутренними import-связями проекта.
- после разбиения `MemoryManager` текущий рабочий контур проекта сохранён.

## 11. Что проект пока не умеет

- не умеет строить карту вызовов между сущностями кода;
- не умеет планировать решение задач;
- не умеет маршрутизировать действия по модулям;
- не умеет генерировать код в рабочем цикле;
- не умеет безопасно тестировать код в песочнице;
- не умеет безопасно менять сам себя.

Техническое уточнение:
- AST-слой и import-зависимости уже работают на живой базе.

## 12. Следующий логичный шаг

Следующий логичный шаг по текущему состоянию:

`начать строить связи уровня "кто кого вызывает" между сущностями кода`

Что должен дать этот шаг:
- следующий уровень понимания физиологии кода;
- основу для ответа на вопрос, что сломается при изменении функции или метода;
- базу для будущих метрик критичности и архитектурной слабости.

## 16. Обновление карты от 2026-04-02: слой сравнения состояний

### Изменения структуры

#### `data/schema_code_map.sql`
- Дополнено:
  - таблица `state_snapshots`;
  - таблица `entity_metric_snapshots`;
  - таблица `module_metric_snapshots`;
  - таблица `state_comparisons`;
  - таблица `entity_metric_diffs`;
  - таблица `module_metric_diffs`.
- Роль:
  - хранить снимки состояний проекта;
  - хранить snapshot-метрики по сущностям и модулям;
  - хранить дельты между двумя последовательными состояниями.

#### `eva_core/memory_state.py`
- Назначение: memory-операции для снимков и сравнений.
- Для чего нужен:
  - создавать state-наборы;
  - сохранять entity/module snapshot rows;
  - сохранять entity/module diffs;
  - отдавать последнее сравнение и top-дельты.
- Статус: активный внутренний memory-модуль.

#### `eva_core/state_comparator.py`
- Назначение: отдельный аналитический модуль сравнения состояний.
- Для чего нужен:
  - снимать текущее состояние после пересчёта метрик;
  - сравнивать его с предыдущим snapshot;
  - формировать дельты по сущностям и модулям.
- Архитектурная особенность:
  - для идентичности сущностей используется стабильная пара `module_name + entity_qualname`, а не entity-id, который меняется при пересборке AST.
- Статус: активный внутренний аналитический модуль.

### Изменение роли `eva_core/self_monitor.py`

- Теперь после полного пересчёта:
  - AST;
  - import/call-связей;
  - impact-слоя;
  - metric-слоя;
  - запускается отдельный `StateComparator`.
- Архитектурное замечание:
  - `self_monitor.py` не сравнивает состояния сам, а только инициирует отдельный аналитический шаг.

### Что проект теперь умеет дополнительно
